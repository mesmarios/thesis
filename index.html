1. Γενικός Σκοπός της Πτυχιακής

Σκοπός της παρούσας πτυχιακής εργασίας είναι η ανάπτυξη ενός ολοκληρωμένου συστήματος ασφάλειας λογισμικού, το οποίο ενσωματώνεται αυτόματα στη διαδικασία ανάπτυξης κώδικα (CI/CD pipeline) και εντοπίζει κακόβουλο λογισμικό, ευπάθειες και εσφαλμένες ρυθμίσεις ασφαλείας σε container environments — πριν ο κώδικας φτάσει σε production περιβάλλον.

Το σύστημα συνδυάζει τεχνολογίες Distributed Systems, Machine Learning και Cloud Security για να παρέχει αυτοματοποιημένη, scalable και αξιόπιστη ανάλυση ασφαλείας. Κάθε βήμα καταγράφεται σε αμετάβλητο audit log για forensic χρήση.

"Πώς μπορούμε να εντοπίσουμε αυτόματα κακόβουλο κώδικα και ευπάθειες σε container environments, χρησιμοποιώντας ML και distributed processing, πριν αυτά φτάσουν σε χρήστες;"

Σύνδεση με Μαθήματα Σπουδών

  • Parallel & Distributed Systems — Redis task queue, παράλληλη εκτέλεση modules σε containers
  • ML Systems at Scale — Εκπαίδευση και inference ML model σε μεγάλο dataset (EMBER ~1M samples)
  • Cloud Security — Container scanning, Kubernetes security, cloud misconfiguration detection
  • Digital Forensics — Immutable audit log κάθε ανάλυσης με timestamps και hashes

Module 1 — Container Vulnerability Scanner

Χρησιμοποιεί το Trivy (open source tool της Aqua Security) για να σκανάρει κάθε Docker image layer-by-layer. Ελέγχει για:

  • Γνωστές CVEs στα installed packages και dependencies
  • Exposed secrets — API keys, passwords, tokens μέσα στον κώδικα
  • Outdated base images με γνωστές ευπάθειες
  • Misconfigured file permissions και overprivileged processes
# Παράδειγμα Trivy output:
Flask 2.0.1  ->  CVE-2023-XXXX  ->  HIGH     -> update to 2.3.0
OpenSSL 1.1  ->  CVE-2022-XXXX  ->  CRITICAL -> update to 3.0
SECRET FOUND ->  AWS_KEY in /app/config.py line 42

Module 2 — ML Malware Analyzer

Εκπαιδευμένο ML model που αναλύει στατικά τα εκτελέσιμα αρχεία μέσα στο container χωρίς να τα εκτελεί (static analysis). Χρησιμοποιεί το EMBER dataset (~1M Windows PE samples) για εκπαίδευση.

  • File Entropy — υψηλό entropy = packed/encrypted = ύποπτο
  • API Call Sequences — CryptEncrypt, DeleteShadowCopies κλπ
  • Import Table — ποιες system libraries φορτώνει
  • Suspicious Strings — URLs, IPs, commands μέσα στο binary
  • C2 Patterns — επικοινωνία με command & control server
# ML Model Output:
{
  "verdict":    "malware",
  "confidence": 0.94,
  "type":       "ransomware",
  "reasons":    ["high entropy", "shadow copy deletion API",
                 "known C2 server pattern"]
}

Module 3 — Cloud Misconfiguration Checker

Ελέγχει τα Kubernetes manifests και cloud configurations για λανθασμένες ρυθμίσεις:

  • Containers που τρέχουν ως root χωρίς λόγο
  • Missing network policies — ports ανοιχτά χωρίς λόγο
  • Overprivileged RBAC roles
  • Missing resource limits (CPU/memory)

Redis Queue — Parallel Execution

Και τα τρία modules τρέχουν ταυτόχρονα μέσω Celery + Redis task queue. Αντί sequentially (3x χρόνος), τρέχουν παράλληλα — εφαρμογή γνώσης από Parallel & Distributed Systems.

4. Πραγματικό Περιστατικό: Axios Supply Chain Attack (Μάρτιος 2026)

⚠ ΠΡΑΓΜΑΤΙΚΟ ΠΕΡΙΣΤΑΤΙΚΟ — Μάρτιος 2026
Βορειοκορεάτες hackers (UNC1069) παραβίασαν το npm package Axios με 83 εκατομμύρια εβδομαδιαία downloads.

Τον Μάρτιο του 2026, η ομάδα UNC1069 παραβίασε τον npm λογαριασμό του κύριου συντηρητή του Axios και ανέβασε δύο μολυσμένες εκδόσεις (1.14.1 και 0.30.4) που περιείχαν RAT (Remote Access Trojan) ως κρυφό dependency.

Πώς έγινε — Supply Chain Attack

  1. Hack του npm account του developer "jasonsaayman"
  2. Upload μολυσμένων versions με fake dependency "plain-crypto-js v4.2.1"
  3. Το dependency εγκαθιστούσε RAT που επικοινωνούσε με C2 server
  4. Διέγραφε τα ίχνη του για να αποφύγει detection
  5. 83 εκατομμύρια developers το κατέβασαν χωρίς να το ξέρουν

Πώς το σύστημα της πτυχιακής θα το απέτρεπε

  • Module 1 (Trivy): Θα εντόπιζε το "plain-crypto-js v4.2.1" ως unverified/malicious dependency.
  • Module 2 (ML Model): Το postinstall script θα είχε C2 patterns και obfuscated code που ενεργοποιούν το model.
  • CI/CD Pipeline: Το deployment θα μπλοκαριζόταν αυτόματα πριν φτάσει στους developers.
  • Audit Log: Timestamp, hash και λόγος rejection θα αποθηκεύονταν ως αμετάβλητη forensic απόδειξη.
# Τι θα έβλεπε ο investigator στο Dashboard:
SCAN FAILED - axios v1.14.1
  [CRITICAL] Malicious dependency: plain-crypto-js@4.2.1
             ML Confidence: 96%  |  Type: RAT Dropper
             Reason: C2 pattern, obfuscated postinstall
  [HIGH]     Suspicious postinstall script with network calls
  DEPLOYMENT BLOCKED
  Audit: 2026-03-31T02:14:33Z | hash: a3f9... | blocked by: ML+Trivy
Το περιστατικό του Axios αποδεικνύει ότι ακόμα και οι πιο trusted libraries μπορούν να παραβιαστούν. Η πτυχιακή αυτή λύνει ακριβώς αυτό το πρόβλημα.

5. ML Model — Σημαντικές Λεπτομέρειες

Dataset: EMBER (Endgame Malware BEnchmark for Research)

  • ~1 εκατομμύριο Windows PE (Portable Executable) samples
  • Labels: benign / malware (balanced dataset)
  • Δημόσιο, standard benchmark — χρησιμοποιείται σε academic papers
  • Επιπλέον samples από MalwareBazaar για πρόσφατα δείγματα
FeatureΤι δείχνει
File EntropyΥψηλό entropy = packed/encrypted code = ύποπτο
Import TableWindows APIs που φορτώνει (CreateRemoteThread, CryptEncrypt κλπ)
API Call SequencesΣειρά API calls χαρακτηριστική ransomware/trojan
StringsURLs, IPs, registry paths, commands μέσα στο binary
Section HeadersΑσυνήθιστα PE sections ή λάθος permissions
File Size / TypeAnomalies σε size ή mismatch extension vs πραγματικός τύπος

Model Architecture:

  • Baseline: LightGBM / Random Forest (γρήγορο, ερμηνεύσιμο)
  • Advanced: Neural Network με embedding layers για API sequences
  • Output: verdict + confidence score + top reasons (explainability)

Σημαντικό: Το model κάνει static analysis — δεν εκτελεί ποτέ το ύποπτο αρχείο. Αυτό το κάνει ασφαλές και γρήγορο για χρήση σε CI/CD pipeline.

6. CI/CD Pipeline & Kubernetes

Τι είναι το CI/CD Pipeline

CI/CD (Continuous Integration / Continuous Deployment) είναι ένα αυτοματοποιημένο σύστημα που εκτελείται κάθε φορά που ο developer κάνει push κώδικα στο GitHub. Αντικαθιστά τα χειροκίνητα βήματα (tests, builds, deployments) με αυτόματες διεργασίες.

Developer: git push
     |
     v   GitHub Actions ενεργοποιείται αυτόματα
     |
  Step 1: Build Docker Image
  Step 2: Run Security Gate (το project μας)  <-- εδώ μπαίνει η πτυχιακή
  Step 3: Run Unit Tests
  Step 4: Deploy to Kubernetes (μόνο αν PASS)
  Step 5: Notify team

Τι είναι το Kubernetes

Σύστημα orchestration containers. Διαχειρίζεται αυτόματα ποια containers τρέχουν, πόσα instances χρειάζονται και τι γίνεται αν κάτι σπάσει. Για την πτυχιακή χρησιμοποιείται minikube — τοπικό Kubernetes cluster χωρίς κόστος cloud.

8. Γιατί Είναι Σημαντικό Σήμερα

  • Axios npm attack (Μάρτιος 2026): Supply chain attack από Βορειοκορεάτες hackers σε library με 83M weekly downloads — ακριβώς το πρόβλημα που λύνει αυτή η πτυχιακή.
  • XZ Utils backdoor (2024): 2 χρόνια undercover operation για εισαγωγή backdoor σε core Linux library.
  • Log4Shell (2021): CVE-2021-44228 επηρέασε εκατομμύρια servers παγκοσμίως — automated scanner θα το εντόπιζε πριν deploy.
  • Αγορά εργασίας: Εταιρείες όπως Snyk ($8.5B), Aqua Security, Prisma Cloud κάνουν ακριβώς αυτό — το DevSecOps skill είναι από τα πιο ζητούμενα το 2026.
